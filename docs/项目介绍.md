# Vigil — BNB 生态生活助手

> Vigil 把官方 skill、执行链路、主动触达和链上可信通信编织成一个面向 BNB 生态的生活助手。当前仓库公开的重点，是展示判断、执行与证据链路如何在同一系统里闭环；这不等于终端用户需要直接面对底层配置。

---

## Executive Summary

Vigil is a BNB ecosystem living assistant that continuously senses on-chain and off-chain signals, judges whether a user should be interrupted, and delivers outcomes through explainable, paper-first execution paths.

**Four modules, one loop:**

| Module | Role | Status |
|--------|------|--------|
| **Living Assistant** | Signal sensing → LLM triage → 6-level interruption policy → voice/call delivery | Landed, repo-verifiable |
| **Execution Engine** | Cross-DEX arbitrage with 6-dimension cost model, 3-layer risk control, auto paper↔live | Landed, repo-verifiable |
| **Agent-Comm** | Wallet identity + EIP-712 credentials + E2E encryption over BNB Chain calldata | Landed, repo-verifiable |
| **Multi-Channel Delivery** | Telegram + cloned-voice TTS + Twilio phone escalation | Landed, repo-verifiable |

**Key numbers:**
- ~87% noise reduction on sample announcements (80 → 8 notify / 12 digest / 60 skip)
- 6-level attention ladder: `silent` → `call_escalation`
- 3-layer risk control with automatic paper degradation
- 4 official Binance Skills integrated into the main product path
- Voice brief target: ≤15s, ≤3 sentences, cloned voice

**Architecture:** Each module (Agent-Comm, Contact Policy, Voice Brief, Signal Radar, Skill adapters) is designed as a detachable, independently reusable component. Other BNB ecosystem projects can adopt any module without importing the rest of Vigil.

**Verify:** `npm run demo:living-assistant` — runs stable local verification with fixture-driven scenarios.

## 一、项目定位

Vigil 不是一个等命令的 bot，而是一个**住在 BNB 生态里的生活助手**。

它能：
- 24 小时持续监控 Binance 生态信号，自动发现机会和风险
- 用 LLM 判断哪些信息值得打扰你，约 87% 降噪

- 在 paper-first 前提下运行套利与执行策略，并由三层风控约束行为
- Agent 之间通过 BNB 链上铭文建立信任，端到端加密通信

一句话：**这不是单点工具，而是一个把感知、判断、执行、触达和可信通信放进同一闭环的链上生活助手。**

> Vigil 基于 OpenClaw 平台构建，复用平台的通道接入、会话编排、提醒投递与交互闭环能力。对终端用户而言，Telegram / 语音 / 电话就是实际入口 — 用户不需要理解底层的协议层或部署细节。

### 1. 🔗 Agent-Comm — 链上铭文通信协议

Agent 之间怎么互相信任、互相通信？不是靠中心化服务器，而是靠 BNB 链上铭文。

当前多数 Agent 通信方案仍依赖 WebSocket、消息队列或 API 网关，Agent-Comm 选择把区块链本身当作消息总线。这减少了中心化基础设施依赖，并把身份与通信路径绑定到链上可验证对象。

**TL;DR**: 钱包 = 身份，BNB Chain calldata = 消息总线，EIP-712 签名凭证 + secp256k1-ECDH E2E 加密 + 完整连接生命周期 + x402 付费协议。零中心化基础设施依赖。

#### 1.1 设计重点：区块链即消息总线

传统 Agent 通信依赖：WebSocket 服务器、消息中间件（RabbitMQ/Kafka）、API 网关、认证服务器。

Agent-Comm 全部不需要。

```
Agent A → BNB Chain 交易（calldata = 加密信封）→ Agent B 监听链上事件 → 解密
```

消息以交易 calldata 的形式刻在 BNB Chain 上：

- **编码**：加密信封 JSON → UTF-8 → Hex → 作为交易 data 字段发送
- **解码**：监听链上交易 → 提取 calldata → Hex → UTF-8 → JSON → 解密
- **大小限制**：单条消息 ≤ 16KB（`AGENT_COMM_DEFAULT_MAX_MESSAGE_BYTES`）

这意味着：部署在任何地方的 Agent，只要能访问 BNB Chain RPC，就能通信。不需要知道对方的 IP、端口、域名。

> **架构视野：控制面 vs 数据面**
> Agent-Comm 当前运行于控制面（Control Plane）— 处理身份交换、能力协商、密钥派生、授权指令等高价值低频操作，BNB Chain 极低的 Gas 费完全可以支撑。架构设计采用控制面/数据面分离：Agent 在链上完成 v2 握手后，可协商出链下加密通道（WebRTC / 加密 WebSocket），链上做强信任路由，链下做高频传输，链上仅做定期状态 Checkpoint。

#### 1.2 创新性：EIP-712 签名的身份凭证体系

Agent-Comm 设计了一套完整的链上可验证身份凭证体系，包含三种签名工件（Signed Artifacts）：

**ContactCard（名片）**

```
{
  cardVersion, protocols: ["agent-comm/2", "agent-comm/1"],
  displayName, handle, identityWallet,
  transport: { chainId, receiveAddress, pubkey, keyId },
  defaults: { capabilityProfile, capabilities },
  issuedAt, expiresAt,
  proof: { type: "eip712", signer, signature }
}
```

- 用 EIP-712 Typed Data 签名，链上可验证
- `identityWallet` 是长期身份锚点（不可变）
- `receiveAddress` 是传输地址（可轮换）
- `capabilities` 声明支持的命令集（`ping`, `start_discovery`, `connection_invite` 等）

**TransportBinding（传输绑定）**

将身份钱包与传输地址密码学绑定：

```
identityWallet ←[EIP-712 签名]→ receiveAddress + pubkey + keyId
```

确保"这个传输地址确实属于这个身份"，防止地址冒充。

**RevocationNotice（撤销通知）**

支持凭证轮换和撤销：

```
artifactDigest → revokedAt + replacementDigest + reason
```

密钥泄露时可以撤销旧凭证、发布新凭证，不影响身份连续性。

**三者组成 Identity Artifact Bundle** — 一个自包含、可离线验证、可传递的身份证明包。

#### 1.3 端到端加密：secp256k1-ECDH + AES-256-GCM

加密套件标识：`secp256k1-ecdh-aes256gcm-v2`

**密钥协商**

```
sharedPoint = ECDH(myPrivateKey, peerPublicKey)  // secp256k1 椭圆曲线
sharedKey = sharedPoint[1:]                        // 去掉前缀，取 32 字节
```

**加密流程**

```
salt = random(16 bytes)
iv = random(12 bytes)
cipherKey = scrypt(sharedKey, salt, keyLen=32)     // 密钥派生
ciphertext = AES-256-GCM(cipherKey, iv, plaintext) // 认证加密
authTag = GCM.getAuthTag()                         // 完整性标签
payload = salt || iv || authTag || ciphertext       // 拼接输出
```

- **scrypt 密钥派生**：即使共享密钥泄露，每条消息的加密密钥仍然不同（随机 salt）
- **AES-256-GCM**：认证加密，同时保证机密性和完整性
- **密钥清零**：加密完成后立即 `key.fill(0)`，防止内存残留

#### 1.4 双版本信封协议

**Envelope v1（Legacy）**

```
{
  version: 1,
  senderPeerId, senderPubkey, recipient, nonce, timestamp,
  command: { type, schemaVersion },
  x402: { scheme, payer, payee, amount, signature },  // 可选付费
  ciphertext, signature
}
```

**Envelope v2（Contact-Based）**

```
{
  version: 2,
  kex: {
    suite: "secp256k1-ecdh-aes256gcm-v2",
    recipientKeyId,        // 接收方密钥 ID
    ephemeralPubkey        // 临时公钥（前向安全）
  },
  ciphertext              // 加密后的 body
}
```

v2 的 body 解密后包含：

```
{
  msgId: UUID,
  sentAt: ISO8601,
  sender: { identityWallet, transportAddress, cardDigest },
  command: { type, schemaVersion, payload },
  payment: { asset, amount, proof },       // x402 付费
  attachments: { inlineCard }              // 内联名片
}
```

**创新点**：
- v2 使用 `recipientKeyId` + `ephemeralPubkey` 实现前向安全 — 即使长期密钥泄露，历史消息仍然安全
- 双版本兼容：发送方自动检测接收方支持的协议版本，按需降级
- 内联名片：连接请求中可以附带签名名片，一次交易完成身份交换

#### 1.5 完整的连接生命周期

```
发现 → 邀请(connection_invite) → 接受/拒绝(accept/reject) → 确认(confirm) → 通信 → 撤销(revoke)
```

每个阶段都是链上交易，不可篡改、可审计：

| 阶段 | 链上行为 | 状态变迁 |
|------|----------|----------|
| 导入名片 | 离线验证签名 | `imported` |
| 发送邀请 | 链上交易 + 内联名片 | `pending_outbound` |
| 收到邀请 | 监听链上事件 | `pending_inbound` |
| 接受连接 | 链上交易 + 能力协商 | `trusted` |
| 拒绝连接 | 链上交易 + 原因 | `blocked` |
| 撤销信任 | 链上 RevocationNotice | `revoked` |

#### 1.6 x402 付费协议

Agent 之间的通信可以附带支付证明：

```
x402Proof = {
  scheme: "x402",
  network, payer, payee, asset, amount, nonce, expiresAt, signature
}
```

三种模式：`disabled`（不收费）、`observe`（记录但不强制）、`enforce`（必须付费才处理）

这意味着 Agent 可以为自己的服务定价 — 你想用我的发现策略？先付费。这是 Agent 经济的基础设施。

> **设计思路**：x402 扮演"可验证加密发票（Verifiable Invoice）"角色，实现了 `observe`（记录凭证）和 `enforce`（见费放行）的单向逻辑，适用于"一手交钱一手交数据"的场景。Schema 设计支持多签担保（Escrow）和去中心化仲裁等扩展，构建完备的 Agent 商业结算层。

#### 1.7 Relay 中继与容错

交易提交支持双通道：

```
submitMode = "direct" → 直接 RPC 提交
submitMode = "relay"  → 签名后通过 Relay 服务提交
```

容错策略：主通道失败自动切换备用通道。Relay 支持超时控制（默认 10s）。BNB Smart Chain 本身具备高吞吐与低 Gas 特性，控制面操作的链上延迟在实际运行中极少构成瓶颈；即便出现异常，已建立连接的数据面通信和 Living Assistant 的信号感知与投递链路均不经过链上，不受影响。

### 2. 💰 套利执行引擎 — 从发现到执行的闭环设计

跨 DEX 价差捕捉，从发现到执行的完整闭环。消费官方 Skill：`binance/spot`（市场数据）、`binance/assets`（持仓就绪度）。

> **定位：信息差套利（Information Arbitrage），不是延迟内卷。** 不参与纳秒级高频竞争 — TS 脚本拼不过 C++/Rust 直连节点的 MEV 机器人。核心思路是：引擎与 Living Assistant 的 Signal Radar 联动，当 LLM 解析出官方公告中的非结构化信号时，尽早将信息转化为执行信号。三层风控体系（特别是 pFail 中的 MEV 惩罚因子和 Expected Shortfall）确保一旦检测到被夹击风险，立即自动降级到 Paper 模式。目标不是抢最快的单，而是用稳健姿势抓取长尾机会。

**TL;DR**: 六维成本模型（手续费/滑点/延迟/MEV/Gas/尾部风险）+ 三层风控（准入门控→熔断器→动态阈值）+ 自动 paper↔live 切换。默认 paper-first，只有风险调整后净边际超过阈值才执行。

#### 2.1 核心算法：跨 DEX 价差捕捉

**机会发现（Scan）**

从多个 DEX 实时获取报价，按 ask 价排序找最低买入价，按 bid 价排序找最高卖出价：

```
sortedByAsk = quotes.sort(ask ASC)  → buy = sortedByAsk[0]
sortedByBid = quotes.sort(bid DESC) → sell = sortedByBid[0]
```

过滤条件：买卖必须来自不同 DEX，且 `sell.bid > buy.ask`（存在正价差）。

**毛边际计算（Gross Edge）**

```
grossEdgeBps = ((sell.bid - buy.ask) / buy.ask) × 10,000
```

单位为 bps（基点，万分之一），直观反映价差幅度。

**流动性估算**

通过买卖盘口价差反推流动性深度：

```
buySpreadBps  = ((buyAsk - buyBid) / buyAsk) × 10,000
sellSpreadBps = ((sellAsk - sellBid) / sellAsk) × 10,000
combinedSpread = avg(buySpreadBps, sellSpreadBps)
liquidityUsd = clamp(1,000,000 / combinedSpread, 25K, 2M)
```

价差越小 → 流动性越深 → 滑点越低。

#### 2.2 六维成本模型

不是简单的"收益 - 手续费"，而是一个考虑六个维度的精细成本模型：

| 维度 | 计算方式 | 说明 |
|------|----------|------|
| 双边手续费 | `takerFeeBps × 2` | 买入 + 卖出各收一次 |
| 双边滑点 | `estimateSlippage() × 2` | 基于仓位/流动性/波动率的非线性模型 |
| 延迟惩罚 | `edgeBps × 0.01 × (latencyMs / 100)` | 延迟越高，价差衰减越快 |
| MEV 惩罚 | 固定 `mevPenaltyBps` | 链上抢跑风险 |
| Gas 费用 | `gasBuyUsd + gasSellUsd` | 双边链上 gas |
| 尾部风险 | `pFail × (totalCost + tailMoveUsd)` | Expected Shortfall |

**滑点估算模型（非线性）**

```
slippage = baseBps + (notional / liquidity)^0.5 × 10 × scale × (1 + volatility)
```

采用平方根冲击模型（square-root impact model），这是做市商和量化交易中的标准方法 — 交易量对价格的冲击与交易量的平方根成正比，而非线性。

**净边际计算**

```
netEdgeBps = grossEdgeBps - feeBps - slippageBps - latencyPenaltyBps - mevPenaltyBps
netUsd = notionalUsd × grossEdgeBps / 10,000 - totalCostUsd
```

#### 2.3 风险调整模拟器

在执行前，模拟器会进行完整的风险调整评估：

**失败概率估算（pFail）**

```
pFail = baseFail(3%) + latencyRisk + edgeRisk + volRisk
```

- `latencyRisk`：延迟越高，执行失败概率越大（最高 +40%）
- `edgeRisk`：净边际越低，越容易被市场波动吃掉（最高 +35%）
- `volRisk`：波动率越高，价格在执行期间变化越大（最高 +20%）

**Expected Shortfall（尾部风险）**

```
ES = pFail × (totalCost + tailMoveUsd)
tailMoveBps = 6 + downsideBps×0.8 + volatilityTail + latencyTail + liquidityTail + slippageTail - edgeRelief
```

综合考虑波动率尾部（最高 130bps）、延迟尾部（最高 20bps）、流动性尾部（最高 25bps）、滑点尾部（最高 20bps），减去正边际带来的缓冲。

**最终判定**

```
latencyAdjustedNetUsd = netUsd - expectedShortfall
riskAdjustedNetEdgeBps = (latencyAdjustedNetUsd / notionalUsd) × 10,000
pass = riskAdjustedNetEdgeBps ≥ threshold(mode)
```

Paper 模式阈值 45bps，Live 模式阈值 60bps。只有风险调整后的净边际超过阈值才会执行。

#### 2.4 三层风控体系

**第一层：准入门控（Live Gate）**

从 Paper 模式升级到 Live 模式必须满足全部条件：

| 条件 | 阈值 |
|------|------|
| 24h 模拟净收益 | > 0 |
| 24h 模拟胜率 | ≥ 55% |
| 连续失败次数 | < maxConsecutiveFailures |
| 24h 权限失败 | = 0 |
| 24h 拒单率 | ≤ 动态阈值 |
| 24h 平均延迟 | ≤ 动态阈值 |
| 24h 滑点偏差 | ≤ 动态阈值 |

**第二层：熔断器（Circuit Breaker）**

运行中持续监控，任一条件触发即自动降级到 Paper：

- 连续失败超限
- 日内回撤超过余额的 1.5%
- 权限失败累计 ≥ 2
- 拒单率 / 延迟 / 滑点偏差超过动态阈值

**第三层：动态阈值（Market-Adaptive）**

风控阈值不是固定的，而是根据市场状态动态调整：

```
marketStress = volatilityStress×0.5 + gasStress×0.3 + liquidityStress×0.2
```

市场压力越大 → 阈值越严格 → 更容易触发熔断。这意味着在极端行情下，系统会自动收紧风控，而不是继续冒险。

#### 2.5 执行引擎（Alpha Engine）

完整的策略循环：

```
获取报价 → 发现机会 → 去重过滤 → 成本评估 → 模拟验证 → 执行/Paper → 记录结果 → 风控检查
    ↑                                                                              ↓
    └──────────────────────── 循环（可配置间隔） ←──────────────────────────────────┘
```

- **报价去重**：相同机会在 TTL 内不重复评估，除非边际变化超过阈值
- **双模式执行**：Paper 模式用模拟结果，Live 模式调用链上执行（quote → swap → simulate → broadcast）
- **自动模式切换**：满足 Live Gate 条件自动升级，触发熔断自动降级
- **状态持久化**：交易记录、余额、模式状态持久化到 SQLite

平时挂着默默跑，不需要人盯。这是助手的"赚钱能力"。

</details>

### 3. 📡 Living Assistant — 主动感知 + 智能判断

## 四、给 BNB 生态的三个可复用贡献

Binance Skills Hub 解决了"Agent 怎么获取数据和执行交易"，但没解决三个关键问题：

### 贡献 1：Agent 之间怎么互相信任？

→ **Agent-Comm 链上铭文协议**

钱包身份 = Agent 身份，BNB 链上刻铭文建立信任，端到端加密通信。这不是私有协议，这是一个可以被整个 BNB Agent 生态复用的信任基础设施。

### 贡献 2：Agent 怎么判断什么时候该打扰用户？

→ **Contact Policy Engine + 6 级注意力阶梯**

Skills Hub 里的 skill 都是被动的 — 你调它才响应。但一个持续运行的助手需要主动判断。这个判断引擎补的是 Skills Hub 生态里的"主动决策层"。

### 贡献 3：Agent 怎么像人一样联系用户？

→ **多渠道投递 + One-Breath Voice Brief Protocol**

不是把数据吐出来，而是用你自己的声音、15 秒内、3 句话讲清楚。克隆音色、电话升级、智能降噪 — 这是 Skills Hub 生态里缺失的"表达层"。

### 架构特性：Skill 化、可拆卸、可独立复用

上面三个贡献不只是"Vigil 内部用了"，它们在架构上就是按可拆卸方式设计的：

| 模块 | 可独立使用？ | 说明 |
|------|-------------|------|
| Agent-Comm 协议 | ✅ | 独立的 `agent-comm/` 目录，自带钱包管理、连接生命周期、加密套件，不依赖 Vigil 其他模块 |
| Contact Policy Engine | ✅ | 输入 `NormalizedSignal`，输出中断级别，不绑定特定信号源或投递渠道 |
| Voice Brief Protocol | ✅ | 输入文本，输出音频文件 + 投递动作，TTS provider 可替换 |
| Signal Radar | ✅ | 适配器模式，每个信号源是独立适配器，新增信号源不改核心 |
| Skill 适配器层 | ✅ | 统一 `SkillAdapter` 接口，~100 行实现一个新适配器，即插即用 |

这意味着：
- **其他 BNB 生态项目可以单独拿走 Agent-Comm** 做 Agent 间信任通信，不需要引入 Vigil 的其他部分
- **其他助手类项目可以单独复用 Contact Policy** 做中断决策，接自己的信号源和投递渠道
- **Vigil 自身的扩展也是拆卸式的** — 加一个新 Skill、新信号源、新投递渠道，都不需要改核心引擎

这不是事后拆分的，而是从第一天就按这个粒度设计的。代码目录结构直接反映了这个边界：

```
src/skills/
├── agent-comm/          ← 独立协议层

│   ├── signal-radar/    ← 可插拔信号源
│   ├── contact-policy/  ← 可独立复用的判断引擎
│   ├── delivery/        ← 可替换的投递适配器
│   ├── voice/           ← 可替换的语音协议
│   ├── llm/             ← 可替换的 LLM 审阅层
│   └── tts/             ← 可替换的 TTS provider
├── module/adapters/     ← Skill 适配器（每个 ~100 行）
└── runtime/             ← 执行引擎 + 风控
```
